# Manage multiple agents with agent view

> Dispatch and manage many Claude Code sessions from one screen. Agent view shows what every session is doing and which ones need your input.

Agent view, opened with `claude agents`, is one screen for all your background sessions: what's running, what needs your input, and what's done. Dispatch new sessions, watch their state at a glance instead of scrolling through transcripts, and step in only when one needs you. Each background session is a full Claude Code conversation that keeps running without a terminal attached, so you can open it, reply, and leave whenever you want.

Use agent view when you have several independent tasks Claude can work on without you watching every step. Dispatch a bug fix, a PR review, and a flaky-test investigation as three rows, keep working in another window, and check back when a row shows it needs you or has a result. To work more directly in any agent's session, attach to the row.

> **Note:** Agent view is in research preview and requires Claude Code v2.1.139 or later (`claude --version`). The interface and keyboard shortcuts may change.

## Quick start

1. **Open agent view** — `claude agents`. Input at the bottom, table fills in as sessions start. `Esc` returns to your shell; sessions keep running while away.
2. **Dispatch a session** — type a prompt + Enter; a background session starts as a row (working / waiting / done), using the model in the header and the directory's permission mode. Every prompt starts its own new session (parallel), each using subscription quota independently.
3. **Peek and reply** — select a row, `Space` opens the peek panel (most recent output / the question it's waiting on); type + Enter to reply without leaving agent view.
4. **Attach and detach** — `Enter` or `→` attaches (full interactive session); `←` on an empty prompt detaches.
5. **Bring an existing session in** — `/bg` inside it, or `←` on an empty prompt to background + open agent view in one step.

You can use `claude agents` as your primary entry point instead of `claude`.

## Monitor sessions with agent view

`claude agents` takes over the full terminal and lists every session grouped by state (pinned + needs-input at top), each row showing name, current activity, and time since last change. By default it shows every background session across all projects; narrow with `claude agents --cwd ~/projects/my-app` (v2.1.141+). Interactive sessions in other terminals don't appear until backgrounded. Subagents and teammates a session spawns aren't listed as separate rows.

### Read session state

| State       | Icon shows as | What it means                                                            |
| :---------- | :------------ | :----------------------------------------------------------------------- |
| Working     | Animated      | Claude is actively running tools or generating a response                |
| Needs input | Yellow        | Claude is waiting on a specific question or permission decision from you |
| Idle        | Dimmed        | The session has nothing to do and is ready for your next prompt          |
| Completed   | Green         | The task finished successfully                                           |
| Failed      | Red           | The task ended with an error                                             |
| Stopped     | Grey          | The session was stopped with `Ctrl+X` or `claude stop`                   |

Icon shape shows whether the process is running: `✻`/animated `✽` = alive; `∙` = process exited (you can still peek/reply/attach, Claude restarts from where it left off); `✢` = a `/loop` session sleeping between iterations (shows run count + countdown). A `PR #N` label at the right edge is the PR the session opened. The terminal tab title shows the awaiting-input count. Background sessions don't need a terminal open — a separate supervisor process runs them. State persists on disk through auto-updates, supervisor restarts, and sleep.

### Row summaries

The one-line summary is generated by a Haiku-class model (refreshes at most once every 15s while working, plus once per turn end). From v2.1.161 a `done/total` count (e.g. `2/5`) appears before the summary when running parallel work items. Each refresh is one short Haiku-class request billed under the session's data-usage terms; on third-party providers it falls back to the session's main model unless `ANTHROPIC_DEFAULT_HAIKU_MODEL` is set.

### Pull request status

A `PR #1234` label appears at the right edge when a session opens a PR (linked in terminals supporting hyperlinks; persists across follow-ups). Multiple PRs show a count (e.g. `3 PRs`). Color: Yellow = waiting on checks/review or checks failed; Green = passed, no blocking review; Purple = merged; Grey = draft/closed.

### Peek and reply

`Space` opens the peek panel (what it needs, recent output, PRs opened). From v2.1.161 it names the longest-running parallel work item. Type + Enter to reply; multiple-choice questions show options (press a number); `Tab` fills a suggested reply; prefix `!` to send a Bash command. `↑`/`↓` peek adjacent sessions; `→` attaches. Voice dictation (v2.1.145+) works in the reply/dispatch inputs.

### Attach to a session

`Enter` or `→` attaches (full interactive session; Claude posts a recap). Attached sessions render fullscreen (no scrollback); scroll with PgUp/PgDn/wheel, `Ctrl+O` for transcript mode. `←` on an empty prompt detaches (or `Ctrl+Z` if a dialog has focus). `Ctrl+C` keeps interrupt behavior; double `Ctrl+C` on empty prompt detaches. Detaching never stops a session; to end from inside, `/stop`. `←` on an empty prompt works from any session (backgrounds it + opens agent view); toggle via `leftArrowOpensAgents` in `/config`.

### Organize the list

Groups: `Ready for review` + `Needs input` above `Working` + `Completed` (these group names don't map 1:1 to states — `Ready for review` = open PR; `Completed` collects finished/failed/stopped). `Ctrl+S` groups by directory. Within a group: `Ctrl+T` pin (keeps process running while idle); `Shift+↑`/`Shift+↓` reorder; `Ctrl+R` rename; Enter on a group header collapses. `Ctrl+X` stops a session; `Ctrl+X` again within 2s deletes it (also deletes a Claude-created worktree incl. uncommitted changes — push first). Older completed sessions fold into `… N more`.

### Filter sessions

Type in the dispatch input to filter: `a:<name>` (sessions running the named agent); `s:<state>` (e.g. `s:working`, `s:blocked`); `#<number>` or PR URL (the session on that PR); any other URL (the session whose first prompt contained it).

### Keyboard shortcuts

`↑`/`↓` move; `Enter` attach or dispatch; `Space` peek; `Shift+Enter` dispatch + attach; `→` attach; `Alt+1..9` attach to session 1–9; `Tab` browse subagents / apply suggestion; `Ctrl+S` switch grouping; `Ctrl+T` pin; `Ctrl+R` rename; `Ctrl+G` open dispatch prompt in `$VISUAL`/`$EDITOR`; `Ctrl+X` stop (again to delete); `Shift+↑`/`Shift+↓` reorder; `Esc` close panel/clear/exit; `Ctrl+C` clear (twice to exit); `?` show all shortcuts.

## Dispatch new agents

### From agent view

Type a prompt + Enter (auto-named; rename with `Ctrl+R`). Paste an image to include a screenshot. Prefixes/mentions: `<agent-name> <prompt>` (first word matches a custom subagent → runs it as main agent); `@<agent-name>` (mention anywhere); `@<repo>` (run in that repo); `/<command>` (suggest skills/commands); `! <command>` (background shell job); `#<number>`/PR URL (select existing session); `Shift+Enter` (dispatch + attach). `/exit`, `/quit`, `/logout` run in agent view itself; every other command/skill is sent to a new background session.

#### Dispatch to a specific directory

A new session runs in the directory agent view was opened from. Target another: open `claude agents` there; or open in a parent dir and mention `@<repo>`; or `cd` + `claude --bg "<prompt>"`.

### From inside a session

`/background` (alias `/bg`) moves the current conversation to a background session (pass a prompt for one more instruction). Backgrounding starts a fresh process resuming from the saved conversation; running subagents/monitors/background commands do NOT transfer (Claude confirms first). Config flags carry through: `--mcp-config`, `--strict-mcp-config`, `--settings`, `--add-dir`, `--plugin-dir`, `--fallback-model`, `--allow-dangerously-skip-permissions`, plus `/add-dir` dirs. Carrying `--allow-dangerously-skip-permissions` keeps `bypassPermissions` reachable but still requires the one-time interactive acceptance.

### From your shell

`claude --bg "investigate the flaky test"`; run a subagent as main agent `claude --agent code-reviewer --bg "…"`; set a name `claude --bg --name "flaky-test-fix" "…"`. Prints short ID + management commands (`claude agents`, `claude attach <id>`, `claude logs <id>`, `claude stop <id>`).

#### Run a shell command

`!` as the first char of the dispatch input runs a shell job (or `claude --bg --exec 'pytest -x'`). PTY-backed; appears as a row; output in memory only, cleans up ~5 min after exit.

### How file edits are isolated

Every background session starts in your working directory; before editing, Claude moves it into an isolated git worktree under `.claude/worktrees/`. Skipped when: already inside a linked worktree; not a git repo and no `WorktreeCreate` hook; the write is outside the working directory. Turn off via `worktree.bgIsolation: "none"` in `.claude/settings.json` (v2.1.143+). Deleting a session in agent view removes a Claude-created worktree (incl. uncommitted changes); `claude rm` keeps one with uncommitted changes and prints its path. A subagent inherits the session's working dir/worktree unless given `isolation: worktree`.

### Set the model

The header model is the dispatch default (from the `model` user setting / `/model`). Override for the whole agent view with `--model`; per session via `claude --bg --model`, attaching + `/model` + `s`, or a subagent's `model` frontmatter.

### Permission mode, model, and effort

A background session reads settings from its directory. `/bg` or `←` keep the current permission mode; dispatching from the input / `claude --bg` uses the directory's `defaultMode` or the subagent frontmatter's `permissionMode`. Mode/model/effort persist across supervisor stop/restart. Set defaults when opening: `claude agents --permission-mode plan --model opus --effort high`. `--agent` sets the default subagent. `bypassPermissions`/`auto` are refused until accepted once interactively. (Flags added across releases: `--permission-mode`/`--model`/`--effort`/`--dangerously-skip-permissions` v2.1.142; `--allow-dangerously-skip-permissions` v2.1.143; `--agent` + honoring `agent` setting v2.1.157.)

### Settings, plugins, and MCP servers

Agent view accepts the same config flags as `claude` (v2.1.142+), applied to agent view + passed to dispatched sessions: `--settings`, `--add-dir`, `--plugin-dir`, `--mcp-config`, `--strict-mcp-config`. Repeat each flag once per value (space-separated form not supported).

## Manage sessions from the shell

| Command                      | Purpose                                                                 |
| :--------------------------- | :---------------------------------------------------------------------- |
| `claude agents`              | Open agent view                                                         |
| `claude agents --cwd <path>` | Scope to sessions started under `<path>`                                |
| `claude agents --json`       | Print active sessions as JSON (add `--all` for completed)               |
| `claude attach <id>`         | Attach to a session in this terminal                                    |
| `claude logs <id>`           | Print the session's recent output                                       |
| `claude stop <id>`           | Stop a session (also `claude kill`)                                     |
| `claude respawn <id>`        | Restart a session with its conversation intact (also `--all`)           |
| `claude rm <id>`             | Remove a session from the list (worktree handling as above)             |
| `claude daemon status`       | Print supervisor state, version, socket dir, worker count               |
| `claude daemon stop --any`   | Stop the supervisor + its sessions (`--keep-workers` to leave them up)  |

## How background sessions are hosted

### The supervisor process

A per-user supervisor process hosts background sessions, separate from your terminal/agent view; it starts automatically and authenticates with the same credentials (no extra network connections beyond the model API). Each background session is its own process. Once a session finishes and sits unattached ~1 hour, the supervisor stops its process to free resources (pinned sessions exempt); state stays on disk and a fresh process restarts on next attach/peek/reply. The supervisor watches the installed binary and restarts into new versions after the auto-updater replaces it (local file watch; detached sessions keep running).

### Where state is stored

Under your Claude Code config dir (or `CLAUDE_CONFIG_DIR`): `~/.claude/daemon.log`; `~/.claude/daemon/roster.json`; `~/.claude/jobs/<id>/state.json`; `~/.claude/jobs/<id>/tmp/`. Each session has `CLAUDE_JOB_DIR` set. Inspect via `claude daemon status` (also in `/doctor`).

### Turn off agent view

Set `disableAgentView: true` or the `CLAUDE_CODE_DISABLE_AGENT_VIEW` env var (administrators can enforce via managed settings).

## Troubleshooting

* **`claude agents` lists subagents instead of opening** — agent view unavailable in your env (e.g. older versions via Bedrock/Vertex/Foundry); `claude update`.
* **Opens with no sessions** — onboarding hint; type a prompt to dispatch.
* **Cannot open agents because work is running** — `←` won't abandon in-flight work; `/tasks` then `/bg` to confirm.
* **Prompt rejected as too short** — describe the task (≥4 chars).
* **Sessions show as failed after shutdown** — shutdown stops sessions; attach/peek/reply restarts from where it left off. Sleep alone doesn't cause this.
* **Background service did not respond** — supervisor stalled; `claude daemon stop --any --keep-workers` to restart while keeping sessions.
* **Cannot read Desktop/Documents/Downloads on macOS** — grant access in System Settings > Privacy & Security > Files and Folders (or Full Disk Access).
* **Slow to respond after attaching** — stopped idle process restarting; pin with `Ctrl+T`.
* **`.claude/worktrees/` filling up** — `git worktree list` + `git worktree remove <path>`.

## Limitations

Research preview: rate limits apply (parallel sessions multiply quota use); sessions are local (preserved across sleep, stop on shutdown); Claude-created worktrees are deleted with the session in agent view (push/merge first).
