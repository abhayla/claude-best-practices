# /provenance-report — per-project "Behind the Scenes" report

Renders a single self-contained HTML report at `docs/process-report/index.html` in
the CURRENT project, showing the work's provenance for the owner. Per-project by
design — each project has a different process, so each report differs.

**Honest scope:** the live sequence (which agent/skill ran when) lives in the
session, not fully on disk. So this report is **authored at session end from real
on-disk data + the session's known sequence** — most accurate when run while the
build is fresh. Auto-harvest from `loop-engineering` state/events is a future upgrade.

## STEP 1: Harvest the real, on-disk data
- `git -C <project> log --oneline` (+ stat) → commits / what changed when.
- `.claude/learnings.json` → decisions, gaps, hub-ward signals captured.
- `.workflows/*/state.json` + `events.jsonl`, `.pipeline/state.json` (if present) → orchestrated step log.
- `test-results/*.json`, `test-evidence/<run>/` → tests + visual verdicts.
- `docs/**` (CONTEXT / VALUE / PRD / RETROSPECTIVE / design records) → requirements, value, decisions.
- The `.claude/` patterns the project uses (skills/agents/rules/hooks) → which were relevant.
Cross-check on-disk data against the session's known sequence; NEVER fabricate steps/numbers.

## STEP 2: Assemble the sections
1. **Summary** — outcome + KPI tiles (tests, build, commits, agents, design options).
2. **Timeline / sequence** — ordered steps with what was caught/corrected.
3. **Roles engaged** — PM/BA · UX · Full-Stack · QA · DevOps (per `engineering-roles.md`), one line each.
4. **Patterns used (how & when)** — skills / rules / hooks / workflows actually used + when.
5. **Agents dispatched** — agent · count · purpose.
6. **Gates & decisions** — G1 UI / G2 acceptance / G3 deploy status; key decisions + assumptions.
7. **Done vs Open** — two columns; open items honest (amber/red).
8. **Loophole / edge-case audit** — checklist with status chips (handled / fixed / open).
9. **Footer** — "Generated by /provenance-report — per-project artifact."

## STEP 3: Render self-contained HTML
Write `docs/process-report/index.html`: single file, ALL CSS inline, icons/charts as
inline SVG/unicode — NO CDN / JS lib / web fonts / network (renders offline). Light,
scannable, responsive. Use ONLY harvested + known facts; flag OPEN/NOT-DONE honestly.

## STEP 4: Verify + report
Render it (serve locally + screenshot, or open) to confirm it's not broken. Print the
path. With `--open`, open it in the browser. Do NOT commit unless asked.

## CRITICAL RULES
- MUST write the report into the TARGET project (`docs/process-report/`), NEVER into the hub — it's per-project.
- MUST be a single self-contained offline HTML (inline CSS + inline SVG; no network).
- MUST use ONLY real on-disk data + the session's known sequence — NEVER fabricate steps, numbers, or pattern usage; flag uncertain/sequence-reconstructed items.
- MUST show OPEN / NOT-DONE / loophole items honestly (not just successes) — the report's value is showing what was NOT done too.
- MUST include the loophole / edge-case audit section (this doubles as the process self-review).
